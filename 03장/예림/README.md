# 03장. 문서 초안 만들기

## 📌 성공적인 문서 작성을 위해 준비하기

### 1. 문서 작성 도구 선택하기
- 문서 작성 도구를 선택할 때에는 두 가지 요인을 고려해야 한다.
  - 콘텐츠의 포맷
  - 초안의 공유 가능 여부

- 작업하기 가장 편안한 도구를 사용해라. 문서를 작성하려고 새로운 도구를 배울 필요는 없다.
- 도구를 선택하는 데 너무 지체하지 말라. 대부분의 경우 기존 워크플로라도 충분하다.

### 2. 빈 페이지 극복하기
- 문서의 가장 앞 부분에 이미 수집한 정보를 나열하여 문서를 시작할 수 있다.
  - 독자
  - 목적
  - 콘텐츠 패턴
- 예시
  - 독자 : REST API를 사용하는 방법을 알고 Corg.ly를 사용하는 개발자
  - 목적 : 분석을 위해 오디오 파일을 Corg.ly 서비스로 업로드하는 방법을 설명
  - 콘텐츠 패턴 : 절차 가이드
 
### 3. 문서 제목과 목표 정의하기
- 제목은 문서의 목적을 가장 간결하고 명확하게 표현해야 한다.
  - 문서를 읽는 목적을 요약해야 한다
  - 예시) Corg.ly에 오디오 파일 업로드하기
- 문서의 목표를 한 가지로 제한하라.


## 📌 개요 작성하기
필요한 모든 단계를 적되, 순서가 맞는지는 따지지 않는다.
### 1. 독자의 기대 충족하기
- 독자가 알아야 하는 것과 해야 하는 것이 무엇인지 생각한다.
- 작업을 계층적으로 그룹화하고, 너무 복잡한 작업은 여러 개로 나눈다.
### 2. 개요 완성하기
- 독자를 염두에 두고 문서 개요를 검토한다.
  - 독자가 알아야 할 추가적인 정보가 있는가?
  - 건너뛰거나 완전히 설명하지 않는 단계까 있는가?
  - 순서대로 읽었을 때 이해가 잘 되는가?
 
## 📌 초안 작성하기
- 초안은 개요에 쓴 주제에 따라 독자를 안내하면서 독자가 필요로 하는 자세한 저보로 각 주제를 확장하는 것이다.
- 내용을 채울 때 제목, 단락, 절차, 목록, 안내 문구를 사용할 수 있다.
### 1. 제목
- 제목은 문서 내의 콘텐츠를 조직화한다.
- 제목을 만들 때 요령
  - 간결하고 명확하고 구체적이어야 한다.
    - 독자가 제목을 빨리 훑어보고 문서의 상위 수준에서 이해할 수 있어야 한다.
  - 가장 중요한 정보를 앞에 배치한다.
  - 각 절에 대해 고유한 제목을 사용한다.
  - 일관성을 유지한다.
### 2. 단락
- 가능하면 한 단락은 문장 5개를 넘지 않도록 구성한다.
### 3. 절차
- 절차의 시작 부분에는 이루고자 하는 목표를 설명함
- 절차의 끝 부분에는 사용자가 절차를 올바르게 수행했는지 확인하는 방법을 덧붙인다.
  - 일종의 단위 테스트 역할
- 절차를 작성할 때 독자가 로그인된 상태여야 하는지 등 시스템의 시작 상태를 확인한다.
- 절차의 각 단계에는 한 가지 동작만 나와야 한다.
- 독자에게 그들이 절차를 제대로 완료했음을 확인하기 위핸 방법을 제공한다.
  - ex) 업로드가 성공했다면 확인 메시지를 받게 될 것이라는 걸 알린다.
 ### 4. 목록
 - 다음과 같은 항목을 목록으로 작성한다.
   - 예시 목록
   - 설정 방법
   - 관련 주제
 - 가장 도움이 되는 방식으로 항목을 나열한다.
 - 목록이 너무 길어지면 훑어보기 불편하므로 항목이 10개가 넘는다면 더 작은 항목으로 구성된 목록으로 나누는 것을 고려
### 5. 안내 문구
- 콜아웃
- 독자가 놓치면 안되는 중요한 정보에 한해 사용 (지속적으로 사용하면 피로를 느낄 수 있음)

## 📌 훑어보기를 고려하여 문서 작성하기
- 기술 문서 독자에 대한 두 가지 근본적이고 역설적인 사실
  - 독자는 정보를 찾기 위해 문서를 본다.
  - 독자들은 여러분이 쓴 글을 매우 조금 읽는다.
- 독자가 콘텐츠를 훑어보고 알맞은 정보를 찾을 수 있도록 문서를 작성하라.
- 콘텐츠를 보다 쉽게 훑어볼 수 있는 여러가지 전략이 있다.
### 1. 가장 중요한 정보를 먼저 제시하자.
- 제목은 문서의 목표를 요약한다.
- 처음 두세 단락에는 중요한 정보를 담는다.
- 작업 절차의 경우 문서를 읽고 나면 무엇을 달성할 수 있는지 알려준다.
### 2. 분량이 많은 텍스트는 나누자
- 여러 개의 긴 단락을 하위 섹션으로 구분
- 목록, 샘플 코드 또는 그래픽 요소로 바꿔서 훑어보기 더 쉽게 만들자.

### 3. 긴 문서는 나누자
- 긴 문서 하나에 너무 많은 독자를 위해 너무 많은 목표를 달성하려 하지 마라.
### 4. 단순함과 명확함을 추구하자
- 짧고 간결한 문서는 아름답다. 문서 초안을 작성할 때 다음과 같이 자문하라.
  - **이 콘텐츠가 독자의 니즈를 충족하는가?**
- 프로젝트 이력이나 설계 고려 사항 같은 정보는 절차 문서에는 적합하지 않다.
- 개발 이력, 설계 이론, 문제에 대한 해설서는 별도 위치에 두고 다르게 제목을 붙여 적절한 서식을 지정한다.
## 📌 막혔을 떄 빠져 나오기
- 막히는 일은 글쓰기 과정의 일부
- 여러분을 가로막는 것이 무엇인지 파악하면 문제를 해결하고 계속 진행하기 쉽다.
  - 내용이 틀릴까봐 두려운가?
  - 자료를 깊이 있게 살펴볼 시간이 부족한가?
  - 완성된 글이 좋지 않을까 봐 걱정인가?
- 빠져나오는 몇 가지 방법을 소개한다.
### 1. 완벽함을 내려놓자.
- 초안은 완벽할 리 없다.
- 첫 번째 초안의 목표는 배포를 위해 완벽하게 다듬어진 문서를 만드는 것이 아니라 **독자에게 필요한 모든 정보를 적는 것**이다.
- 콘텐츠가 완벽해야 한다는 생각을 놓고, 문법에 대한 걱정을 멈추고, 아이디어를 페이지에 적는 데 집중하라.
### 2. 도움을 청하자
- 다른 사람과 이야기하여 문제를 해결하라
### 3. 누락된 내용은 표시하자
- 중요한 정보가 누락되었다면 이를 표시해두고, **확실히 채울 수 있는 부분에 대해 작업을 계속하라**
- 처음부터 문서를 정확하고 순서에 맞게 작성하는 데 매달리지 마라.
### 4. 꼭 순서대로 작성할 필요는 없다
- 자신에게 가장 잘 맞는 순서대로 작성하라.
### 5. 작성 수단을 바꿔 보자
- 텍스트 편집기로 잘 되지 않는다면 다른 프로그램으로 바꿔보거나 컴퓨터에서 완전히 벗어나 보라.
- 종이에 아이디어를 적거나 화이트보드에 스케치
- 쓰기보다 말하기가 편하다면 음성 받아쓰기를 활용
- 중요한 것은 다양한 도구를 시험해 자신에게 가장 잘 맞는 도구를 찾는 것
## 📌 템플릿으로 작성 시작하기
템플릿을 사용하면 구조보다는 내용에 집중할 수 있어 문서 작성이 쉬워진다.
- 문서 유형이 일반적일 수록 템플릿이 더 유용
- 일반적으로 템플릿화하는 문서
  - 유사한 앱에 대한 절차 가이드
  - API 레퍼런스 및 통합 작업용 참조 문서
  - 릴리스 노트
- 버그 리포트 템플릿
  ```
  **버그 제목**
  **환경**
    기기/OS, 브라우저, 소프트웨어 버전 등
  **재현 절차**
  1.
  2.
  3.
  ...
  9.
  10.
  11.
  12.

  **예상 결과**
  **실제 결과**
  **스크린샷 / 이미지**
  
  ```
## 📌 첫 번째 초안 완성하기
완료 여부를 확인하려면 다음과 같이 자문해보라.
- 문서 제목에 문서의 목표가 요약되어 있는가?
- 섹션 제목들이 모여 문서를 적절히 요약하는가?
- 초안이 처음부터 끝까지 독자의 니즈를 다루는가?
- 정보의 흐름이 이해하기에 적절한가?
- 초안이 마찰로그에서 찾아낸 문제를 다루는가?
- 초안이 문서화 패턴이나 템플릿을 올바르게 따르고 있는가?
- 절차 일부 또는 전체가 작동하는지 테스트하고 확인했는가?

## 📌 요약
- 문서의 대상, 목적, 패턴을 정의하는 것으로 시작하라. 문서의 목표가 문서의 제목에서 드러나야 한다.
- 문서의 개요를 만들고 제목, 단락, 목록, 안내 문구를 사용해 구체화하라.
- 독자는 문서를 훑어볼 것이므로 가장 중요한 정보를 먼저 제시하고 독자를 위해 내용을 나눠 정보를 쉽게 찾을 수 있도록 해라.
- 유사한 문서를 여러 개 만든다면 템플릿을 만들어 사용하라.
