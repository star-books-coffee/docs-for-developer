# 02장. 문서화 계획하기
## 📌 콘텐츠 유형
- 개발자 문서에 일반적으로 나타나는 콘텐츠 유형을 소개한다.
### 1. 코드 주석
- 과거의 내린 결정의 맥락을 남겨두면 훗날 코드를 볼 개발자들이 시간을 많이 절약할 수 있다
- 이에 반대하는 입장도 있지만 코드 주석은 코드가 무슨 일을 하는지에 대한 혼란과 모호성을 줄여주고 정보를 제공한다

### 2. README
사용자가 이 코드가 왜 존재하는지 이해하는 데 도움을 주고자할 때 사용
- README 템플릿
  ```
  코드 상위 수준에서 하는 일을 요약하는 한 두 개의 단락

  ## 설치
  1.
  2.
  3.
  4.
  5.

  ## 사용 예시

  ## 문제 해결

  ## 변경 로그

  ## 추가 자료

  ## 라이선스 정보
  ```
- 간결하고 유익하며 정확하고 **최신 정보**를 담고 있어야 함


### 3. 시작하기 문서
제품의 첫인상 & 첫 번째 사용자 경험으로 안내하는 역할
- 사용자가 제품이 무슨 일을 하고 어떤 일을 할 수 있는 **빨리** 이해하도록 도와주어야 한다
- 시작하기 문서 작성 시 스스로에게 물어야 할 몇 가지 질문
  - 이 서비스가 무엇이며 핵심 기능은 무엇인지 가장 빨리 설명하는 방법은 무엇일까?
  - 이 제품을 설치하고 사용하는 가장 간단한 절차는 무엇일까?
  - 신규 사용자가 가질 수 있는 가장 중요한 질문은?
  - 사용자가 이 서비스로 할 수 있는 멋진 일은?
- 시작하기 문서는 사용자가 이 제품을 토대로 실제로 개발할 수 있도록 이끌어야 한다

### 4. 개념 문서
사용자가 서비스 이면의 개념과 아이디어를 이해하는 데 도움을 주는 문서
- 서비스가 **어떻게** 작동하는지 사용자에게 설명
- 구현 세부 사항이 들어가지 않도록 해야 함 (구현 세부 사항은 절차적 콘텐츠에 속함)
- 회의록, 설계 문서, 다이어그램, 내부용 문서 등이 훌륭한 원재료
- **개념 문서는 짧고 간결하게 유지해야 한다**
- 하나의 문서에서 설명하는 개념의 수를 제한하라
  - 새로운 개념 여러 개를 한꺼번에 설명하면 내용이 금방 복잡해진다
- 개념 가이드 템플릿
  ```
  첫 번째 단락, 문서에서 설명하는 개념을 소개함.

  **개요**
  개념이 어떻게 작동하는지에 대한 기술적 개요 제시. 하위 섹션에서 추가적인 하위 구성 요소나 관련 개념 설명

  
  **관련 개념1**
  ...
  **관련 개념2**
  ...

  **추가 자료**
  개념을 구현하는 튜토리얼, How-to 가이드 등 관련된 문서 나열
  
  ```
  
### 5. 절차 문서
- 튜토리얼, How-to 가이드
- 절차 문서는 독자에게 일련의 구조화된 단계를 따라 특정 목표를 달성하는 방법을 보여준다
- 하나의 단계는 사용자가 취해야 하는 한 가지 동작을 설명해야 한다
- 가이드와 튜토리얼을 작성하는 유용한 패턴
  - 한 페이지에 사용자가 필요로 하는 동작을 모두 포함 시켜라 (가이드가 최대한 독립된 형태가 되도록)
  - 단계의 수는 사용자에게 필요한 만큼으로 제한하라 (절차에 단계가 많으면 복잡해 보임 & 유지 관리도 더 필요해짐)
  - 설명이 너무 길어지지 않도록 하라 (일반적인 모니터로 볼 때 두 개 이상의 단계가 화면에 나오도록 절차를 작성하는 것을 권장)
    - 절차에 설명이 너무 많다고 느껴진다면 그 정보를 개념 가이드로 분리
#### 5-1. 튜토리얼
사용자에게 특정 목표를 달성하는 방법을 가르쳐 주는 절차 문서
- 너무 많은 동작을 한 문서에 넣으려고 해선 안된다

  > 아무리 노력해도 긴 튜토리얼을 더 작은 단계로 줄일 수 없다면 서비스 자체가 너무 복잡하다는 뜻일 수도 있다

#### 5-2. How-to 가이드
절차적 콘텐츠의 핵심 유형. 사용자가 서비스로 특정 단계를 수행함으로써 비즈니스 문제를 해결하는 방법을 보여준다
- 사용자를 위한 진정한 차별 요소
  - 사용자가 겪는 문제에 대한 해결책을 구축하도록 도와주는 문서이기 때문
- How-to 가이드를 계획할 때에는 **사용자의 니즈에 주의를 기울이고** 사용자가 무슨 일을 하기를 원하는지에 대한 **회사의 전략 이해**
- How-to 가이드를 작성하는 좋은 패턴은 단순한 단어를 사용하고, 동작을 명확히 설명하고, 가이드로 해결하는 문제를 지속적으로 늘려가는 것
- **시작 부분에는 필수 조건을 포함시켜라**
  - (운영 체제나 패키지 필수 버전 설치와 같은)의존성
  - 전문적인 기술이나 지식이 정말 필요하다면 포함시켜야 하지만 **가급적이면 피해라**
- **효과적인 How-to 가이드는 사용자가 최대한 한 페이지에 머무르도록 한다**
  - 링크를 너무 많이 클릭하면 사용자의 주의를 흐트릴 수 있다
  - 문서 중간에 링크가 있으면 주의를 산만하게 살 수 있다
    => 대신 페이지 끝에 추가 자료에 대한 링크를 제공하라
- How-to 가이드 템플릿
  ```
  첫 번째 단락에서는 핵심 개념을 소개하고 이 가이드에 대한 개괄적인 정보를 제공한다.
  **필수 조건**
  이 가이드에 있는 단계를 따라가기 전에 사용자가 해야할 일들의 목록 제공
  **단계**
  1.
  2.
  3.
  4.
  5.
  ...

  **다음 단계**
  이 가이드에 있는 단계를 수행한 다음에 따라야 하는 추가 문서의 링크 제공
  
  ```

### 6. 참조 문서
개념 문서와 절차 문서가 학습을 돕고 정보를 제공할 목적이라면 참조 문서는 전적으로 **인과관계**, 즉 어떤 동작이 어떤 결과를 낳는지를 다룬다
#### 6-1. API 레퍼런스
사용자가 소프트웨어 구축을 시작하는 데 믿고 참조할 수 있는 자료
- 좋은 API 문서의 특징
  - 모든 리소스와 엔드포인트에 대한 상세한 참조 정보를 제공
  - 풍부한 예제 제공
  - 상태 코드와 오류 메시지를 나열하고 정의
- API에 대한 참조 정보를 제공하는 가장 좋은 방법은 코드에 설명을 담은 주석을 달고 소스 코드로부터 API 레퍼런스를 자동 생성하는 것
#### 6-2. 용어집
특정 서비스, 분야 또는 산업에서 사용되는 용어와 정의를 모은 것
- 용어를 일관성 있게 사용하는 데 도움
- 동일한 용어가 서로 다른 의미로 정의되거나 같은 의미에 대해 서로 다른 용어를 사용하면 사용자에게 실망감을 주게 된다
> 용어집에서는 외부 링크를 제한하라. 외부 링크를 제공하면 콘텐츠를 제3자의 손에 맡기는 셈이다
### 6-3. 문제 해결 문서
사용자가 제품에 대한 문제를 파악하면 문제 해결 문서를 사용해 임시 대처법을 다양한 방식으로 문서화할 수 있다
- 현 상태에서 일단 활용할 수 있는 해법 제공
- 어떤 에지 케이스가 지원되지 않는지 사용자에게 명확하게 알려라
- 너무 많이 설명하기보다는 해결방법에 집중하라
- 오류 메시지를 문서화하는 좋은 패턴은 같은 범주의 메시지들을 한 페이지에 그룹화하는 것
- 메시지가 발생할 가능성이 있는 절차 문서나 튜토리얼에 구체적인 오류 메시지를 포함하는 것이 좋다
- 문제 해결 가이드 템플릿
  ```
  **문제1**
  설명:
  해결 단계:
    1.
    2.
  **문제2**
  설명:
  해결 단계:
    1.
    2.
  ```
> FAQ 형식보다는 사용자의 문제와 해결 방법을 나열하는 것이 좋다. FAQ는 논리와 흐름 없이 정리되지 않은 긴 질문 목록이 되는 경우가 많기 때문이다.
### 6-4. 변경 사항 문서
중대한 변경이나 단절적 변경이 있을 때는 어떤 변경이 언제 왜 일어났는지에 대한 정보를 제공해야 한다
- 변경 사항을 시간순으로 나열하고 다음과 같은 데이터를 넣는다
  - 이전에 지원되던 버전 및 통합 또는 지원 중단된 기능
  - 파라미터나 중요한 필드의 이름 변경
  - 옮겨진 오브젝트나 리소스
- **릴리스 노트**는 다른 유용한 문서 유형이다
  - 어떤 변경이 왜 일어났는지에 대한 맥락을 제공한다
- 릴리스 노트 템플릿
  ```
  2023-03-10
  항목1
    - 요약
    - 영향
    - 이유
    - 필요한 조치
  항목2
  ...

  2023-03-08
  ...
  ```
## 📌 문서화 계획 만들기
1. 문서화 계획을 수립 시 도움이 되는 질문에 답해본다
  - 타깃 고객은 누구인가?
  - 그들이 얻기를 바라는 가장 중요한 이점은?
  - 중요도 순으로 볼 때, 어떤 기능들을 출시할 것인가?
  - 사용자가 제품 출시에서 기대하는 점은?
  - 사용자가 제품이나 기능을 사용하기 전 필요한 지식이 있는가?
  - 어떤 유스 케이스들을 지원하는가?
  - 사용자가 맞닥뜨릴 수 있는 알려진 문제나 마찰 요소가 있는가?
2. 콘텐츠 개요 작성
  - 작성해야 하는 페이지의 제목과 각 페이지의 콘텐츠 유형을 포함하는 목록
- 문서화 계획에 사용를 위한 여정이 일관성 있게 반영되어 있다면 잘 작성했다고 볼 수 있다

## 📌 요약
- 문서화 계획은 만들어야 할 콘텐츠 및 콘텐츠 유형을 문서 작성에 앞서 간략하게 설명하는 문서
- 콘텐츠 유형은 정보를 제시하는 다양한 방법
  - 코드 주석
  - README
  - 시작하기 문서
  - 개념 문서
  - 절차 문서 및 참조 문서
- 문서화 계획은 사용자가 원하는 콘텐츠의 밑그림 역할 & 문서를 작성하는 데 집중할 수 있도록 돕는다
